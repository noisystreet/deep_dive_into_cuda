CICC 分析：CUDA Device 编译器探秘
=====================================

   cicc = CUDA Intermediate Code Compiler，是 CUDA
   工具链中最大的组件（77 MB）， 负责将 CUDA C++ device 代码编译为 PTX
   虚拟汇编。

   分析基于 CUDA 13.1 (build 37061995)

二进制概览
-------------

基础信息
~~~~~~~~~~~~

::

   文件: /usr/local/cuda/nvvm/bin/cicc
   大小: 80,651,848 字节 (≈ 77 MB)
   类型: ELF 64-bit LSB executable, x86-64, version 1 (GNU/Linux),
         dynamically linked, stripped

**发现**: 与 nvcc (PIE) 不同，cicc 是
**ET_EXEC**\ （绝对可执行），而不是 PIE，说明它不参与 ASLR
的地址随机化——典型的性能优先的静态布局编译器设计。

依赖库
~~~~~~~~~~

::

   linux-vdso.so.1
   libpthread.so.0     ← 线程（编译耗时较长，可能使用线程池）
   librt.so.1          ← 实时时钟
   libdl.so.2          ← 动态加载（可能用于加载 libnvvm.so）
   libm.so.6           ← 数学库（浮点常量折叠需要）
   libc.so.6           ← C 标准库

cicc 的依赖非常轻量，它实际的核心功能依赖通过 ``dlopen``
加载额外模块（见后文）。

版本标识
~~~~~~~~~~~~

::

   Generated by NVIDIA NVVM Compiler     ← PTX 输出中的标识
   Compiler Build ID: CL-37061995        ← 编译构建 ID
   Cuda compilation tools, release 13.1, V13.1.115
   Based on NVVM 7.0.1                   ← NVVM IR 版本

NVVM 三层架构
----------------

cicc 内部基于 **NVVM (NVIDIA Virtual Machine)** 框架，分为三层：

.. mermaid:: ../_static/cicc_architecture.mmd

libnvvm.so — 编译核心库
~~~~~~~~~~~~~~~~~~~~~~~~~~~

::

   路径: /usr/local/cuda/nvvm/lib64/libnvvm.so.4.0.0
   大小: 63,660,448 字节 (≈ 61 MB)

暴露了 17 个 API 符号：

::

   nvvmVersion                  ← 版本查询
   nvvmIRVersion                ← IR 版本查询
   nvvmLLVMVersion              ← LLVM IR 版本查询
   nvvmCreateProgram            ← 创建编译程序
   nvvmDestroyProgram           ← 销毁编译程序
   nvvmAddModuleToProgram       ← 添加模块 (LLVM IR)
   nvvmLazyAddModuleToProgram   ← 惰性添加模块 (延迟加载)
   nvvmCompileProgram           ← 编译主入口
   nvvmVerifyProgram            ← 验证程序
   nvvmGetCompiledResultSize    ← 获取输出大小
   nvvmGetCompiledResult        ← 获取编译结果 (PTX)
   nvvmGetProgramLogSize        ← 获取日志大小
   nvvmGetProgramLog            ← 获取编译日志
   nvvmGetErrorString           ← 错误码转字符串
   nvvmDisableConcurrentAPI     ← 禁用并发 API
   __nvvmHandle                 ← 内部句柄
   __nvvmHandleTarget           ← 内部目标句柄

**核心编译流程**:

::

   nvvmCreateProgram → nvvmAddModuleToProgram (多次) → nvvmCompileProgram → nvvmGetCompiledResult

``nvvmCompileProgram`` 是实际的编译入口，它接收 architecture
字符串和优化选项，内部调用 LLVM Pass Pipeline。

libdevice.10.bc — 内置函数库
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

::

   路径: /usr/local/cuda/nvvm/libdevice/libdevice.10.bc
   大小: 454 KB
   格式: LLVM IR bitcode (NVVM IR 7.0.1)
   目标: nvptx64-nvidia-gpulibs

这是 CUDA 数学函数的 **LLVM IR 预编译版本**\ 。它包含了：

-  **数学函数**: sin, cos, tan, exp, log, pow, sqrt, cbrt 等
-  **整数函数**: clz, popcount, abs, min, max 等
-  **特殊函数**: Bessel 函数、误差函数、gamma 函数
-  **CUDA 内置**: ``__syncthreads``\ 、\ ``__threadfence``\ 、原子操作
-  **向量操作**: ``vadd2``, ``vadd4``, ``vabsdiff`` 等 SIMD 向量函数

函数列表（部分）:

::

   __nv_clz, __nv_clzll, __nv_sin, __nv_cos, __nv_exp, __nv_log,
   __nv_pow, __nv_sqrt, __nv_rsqrt, __nv_sincos, __nv_fma,
   __nv_vadd2, __nv_vadd4, __nv_vcmpeq2, __nv_vcmpeq4, ...
   __reduce_add_sync, __reduce_min_sync, __reduce_max_sync, ...

**关键设计**: libdevice 以 LLVM IR bitcode
而非源码形式分发，这样编译器可以在链接时进行内联和常量折叠。注意 target
triple 是 ``nvptx64-nvidia-gpulibs`` 而非
``nvptx64-nvidia-cuda``——它是在编译期间使用的库模块，而不是运行时。

编译流程详解
---------------

内部编译 pipeline
~~~~~~~~~~~~~~~~~~~~~

::

   输入: CUDA C++ device 代码 (已预处理为 .cpp1.ii)
                       │
                       ▼
       ┌──────────────────────────────┐
       │ 1. CUDA 语法 → LLVM IR       │ ← clang-based CUDA frontend
       │   (device 代码提取)           │   (集成在 cicc 内部)
       └──────────────┬───────────────┘
                      │
                      ▼
       ┌──────────────────────────────┐
       │ 2. NVVM IR 生成              │ ← Target: nvptx64-nvidia-cuda
       │   Data layout:               │
       │   e-i64:64-v16:16-v32:32     │
       │   -n16:32:64                 │
       └──────────────┬───────────────┘
                      │
                      ▼
       ┌──────────────────────────────┐
       │ 3. 链接 libdevice.10.bc      │ ← LLVM link time
       │   (加载所需内置函数)          │
       └──────────────┬───────────────┘
                      │
                      ▼
       ┌──────────────────────────────┐
       │ 4. LLVM 优化 Pass Pipeline   │ ← -O0/-O1/-O2/-O3
       │   Module Pass → Function     │
       │   Pass → Loop Pass 迭代       │
       └──────────────┬───────────────┘
                      │
                      ▼
       ┌──────────────────────────────┐
       │ 5. NVVM 专有 Pass           │
       │   - __nvvm_reflect resolution│
       │   - PTX lowering            │
       └──────────────┬───────────────┘
                      │
                      ▼
       ┌──────────────────────────────┐
       │ 6. NVPTX 后端               │
       │   - 指令选择                 │
       │   - 寄存器分配               │
       │   - PTX 发射                │
       └──────────────┬───────────────┘
                      │
                      ▼
       输出: PTX (text format, .ptx)

关键内部机制
---------------

``__nvvm_reflect`` 反射机制
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

这是一个编译期反射接口：

::

   A key=value pair. Replace __nvvm_reflect(name) with value.

例如 ``__CUDA_FTZ`` (flush-to-zero)、\ ``__CUDA_PREC_SQRT`` (精确开方)
等编译选项，通过 ``__nvvm_reflect``
在编译期常量化，使条件分支在编译时就被消除：

.. code:: llvm

   @.str = private unnamed_addr constant [11 x i8] c"__CUDA_FTZ\00"
   %reflect = call i32 @__nvvm_reflect(i8* @.str)
   %ftz_enabled = icmp ne i32 %reflect, 0

cicc 将这些调用替换为常量值，从而死代码消除可以删除未使用的分支。

~~~~~~~~~~~~~~~~~~~~~~~~~~~

从字符串中提取的架构相关错误信息可以反推出 cicc 维护的架构特性表：

   128-bit atomic load/store         : sm_70+ (Volta)
   128-bit atomic exchange/CAS       : sm_90+ (Hopper)

   .alias (PTX 6.3)                  : sm_30+ (Kepler)

地址空间与合法性检查
~~~~~~~~~~~~~~~~~~~~~~~~

cicc 包含完整的 addrspace 检查逻辑：

::

   shared variables can't be initialized     ← 共享内存不能初始化
   texture/surface variables must be global  ← 纹理/表面必须在全局地址空间
   atomic operations on non-i32/i64/i128    ← 原子操作类型限制
   function pointers should always use addrspace(0) ← 函数指针限制

优化示意图
~~~~~~~~~~~~~~

cicc 的 LLVM pass pipeline 大概按以下阶段执行（来自字符串中的 pipeline
扩展点名称）：

::

   Module Pipeline:
     PipelineStart
       → EarlySimplification
         → OptimizerEarly
           → [CGSCC Pipeline]
               → CGSCCOptimizerLate
           → [Function Pass Pipeline]
               → VectorizerStart
                 → ScalarOptimizerLate
                   → LateLoopOptimizations
                     → LoopOptimizerEnd
                       → VectorizerEnd
                         → Peephole
           → FullLinkTimeOptimizationEarly
             → FullLinkTimeOptimizationLast
       → OptimizerLast

总结
-------

cicc 是什么？
~~~~~~~~~~~~~~~~~

cicc 是一个 **基于 LLVM 7.0.1 / NVVM 的 CUDA→PTX
交叉编译器**\ ，内部集成：

+-----------------+-----------------+-----------------+-----------------+
| 组件            | 提供者          | 大小            | 说明            |
+=================+=================+=================+=================+
| cicc 包装器     | NVIDIA          | ~16 MB          | UI、            |
|                 |                 |                 | 命令行、libnvvm |
|                 |                 |                 | API 封装        |
+-----------------+-----------------+-----------------+-----------------+
| libnvvm.so      | NVIDIA (基于    | 61 MB           | LLVM + NVVM     |
|                 | LLVM)           |                 | Pass Pipeline + |
|                 |                 |                 | NVPTX 后端      |
+-----------------+-----------------+-----------------+-----------------+
| libdevice.10.bc | NVIDIA          | 454 KB          | 内置函数 LLVM   |
|                 |                 |                 | IR bitcode      |
+-----------------+-----------------+-----------------+-----------------+

cicc 的独特之处
~~~~~~~~~~~~~~~~~~~

1. **LLVM-based but proprietary**: 基于 LLVM 7.0.1，但加入了 NVIDIA
   专有的 NVVM Pass 和 NVPTX 后端
2. **分层编译**: 通过 libnvvm.so 提供 C API，允许其他工具（如
   nvlink）重用编译能力
3. **内置函数 as LLVM IR**: libdevice 以 LLVM bitcode
   形式分发给编译器，而非运行时链接
4. **Reflect 机制**: 通过 ``__nvvm_reflect``
   将编译选项常量化，实现零开销条件检查
5. **全功能优化 Pipeline**: 支持完整的 LLVM 优化 Pass，从 -O0 到
   -O3，以及自定义 Pass pipeline

~~~~~~~~~~~~~~~~~~~~
